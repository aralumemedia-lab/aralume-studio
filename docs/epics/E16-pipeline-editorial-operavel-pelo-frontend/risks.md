# E16 - Risks

| Risk | Evidence | Why it matters | Mitigation |
| --- | --- | --- | --- |
| Mixed real and mock transport in the app shell | `src/services/api-client.ts` still exports dashboard and agent office data from `mock-api.ts`; `src/routes/dashboard.tsx` and `src/routes/agent-office.tsx` consume those helpers. | A reviewer can mistake a demo surface for an operational surface. | Keep the epic stories on real editorial services and avoid any mock import in the epic flow. |
| No dedicated visual-plan route exists today | `src/routes` has no visual-plan route; the current repo only exposes the visual-plan API contract and backend handlers. | The epic can drift into an undecided UI shape. | Decide the exact frontend placement before implementation; do not code around uncertainty. |
| Editorial audit hooks are absent from the current editorial service | `server/src/modules/editorial/editorial.service.ts` has no audit dependency, while publications and metrics already record audit logs. | The flow would be operable but not fully auditable. | Add the audit requirement to the epic and test it explicitly. |
| Persistence is process-local in the baseline | The editorial repository is in-memory and the current spec documents it as deterministic process-local storage. | Reload proof is possible, but restart durability is not. | Accept reload persistence only in this epic; do not pretend to have a database. |
| Version loss or duplicate ordering | The backend already rejects duplicate version numbers and duplicate scene order in tests, but the frontend flow has not yet proved the whole path. | History can be overwritten or scene order can become unstable. | Keep version and scene ordering tests in the story acceptance criteria. |
| Cross-channel leakage | Editorial entities all carry `channelId`, and the service rejects mismatched links, but the UI must preserve that rule in every action. | A single cross-channel write can contaminate the whole editorial trace. | Keep channel-scoped queries and explicit conflict tests in every story. |
| Scope excess | The surrounding repo includes media, render, clips, approvals, publication, metrics and costs surfaces. | The first remediation epic can easily grow into a broad hardening sprint. | Keep those domains out of scope and refer them to later epics. |
| Visual regression | The app uses a dense SaaS admin design language with multiple states and long texts. | Poor layout at desktop sizes can hide the new flow or break the QA gate. | Require the four desktop viewports and both sidebar states. |
| E2E infra gaps | Browser acceptance is required for this epic. | Without it, the flow may pass tests but not be operable. | Keep route tests plus browser E2E and screenshots as mandatory evidence. |
